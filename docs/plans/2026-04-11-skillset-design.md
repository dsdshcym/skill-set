# Skillset: Multi-Skill Repo Support

## Motivation

Some skills only make sense together — they form a "skill set." Examples: the `extract-notes*` family, or the `superpowers:*` collection. Today each skill is an independent clone, even when multiple skills come from the same repo. Skill-set should treat a group of skills from one repo as a first-class unit.

## Data Model

`Skillset` replaces `Skill` as the core type. A single skill is a special case of a skillset.

```typescript
interface Skillset {
  name: string;       // identifier, also clone dir name
  origin: string;     // git repo URL
  root_path: string;  // base path prefix (default: "")
  skills: string[];   // skill names within the repo
  branch?: string;
  pin?: string;
}
```

Each skill resolves to `{root_path}/{skill_name}` within the repo.

## Skillfile Format

`[[skillset]]` is the primary declaration. `[[skill]]` is backwards-compatible sugar that normalizes to a single-skill skillset at parse time.

```toml
[[skillset]]
name      = "superpowers"
origin    = "https://github.com/someone/superpowers"
root_path = "skills"
skills    = ["brainstorming", "debugging", "tdd"]
# resolves to: skills/brainstorming, skills/debugging, skills/tdd

[[skillset]]
name      = "dotfiles-skills"
origin    = "https://github.com/dsdshcym/dotfiles"
root_path = ".claude/skills"
skills    = ["extract-notes", "extract-notes-review"]

# Backwards-compat sugar — equivalent to a single-skill skillset:
[[skill]]
name   = "tdd"
origin = "https://github.com/anthropics/claude-skills"
path   = "tdd"
# normalizes to: Skillset { name: "tdd", skills: ["tdd"], root_path: "", ... }
```

`root_path` defaults to `""`. With `root_path = ""`, skill `"brainstorming"` resolves to path `"brainstorming"` (top-level directory).

Skills with custom paths can use inline table overrides:
```toml
skills = [
  "brainstorming",
  { name = "debugging", path = "custom/location" }
]
```

## Skillfile.lock Format

Flat `[[skill]]` entries with a `skillset` field linking back to the parent. Auto-generated by `install`/`freeze`, never hand-edited.

```toml
[[skill]]
name     = "brainstorming"
origin   = "https://github.com/someone/superpowers"
path     = "skills/brainstorming"
pin      = "abc1234def5678"
skillset = "superpowers"

[[skill]]
name     = "debugging"
origin   = "https://github.com/someone/superpowers"
path     = "skills/debugging"
pin      = "abc1234def5678"
skillset = "superpowers"

[[skill]]
name     = "extract-notes"
origin   = "https://github.com/dsdshcym/dotfiles"
path     = ".claude/skills/extract-notes"
pin      = "fedcba9876543210"
skillset = "dotfiles-skills"
```

Skills from the same skillset share the same `pin` (same repo, same commit). Installing from the lock file groups by `skillset` to produce shared clones.

## Directory Layout

One clone per skillset. N symlinks per skillset.

```
~/.claude/
├── Skillfile
├── Skillfile.lock
├── skill-repos/
│   ├── superpowers/                  # one clone
│   │   └── skills/
│   │       ├── brainstorming/
│   │       ├── debugging/
│   │       └── tdd/
│   └── dotfiles-skills/              # one clone
│       └── .claude/skills/
│           ├── extract-notes/
│           └── extract-notes-review/
└── skills/                           # flat symlinks
    ├── brainstorming      → ../skill-repos/superpowers/skills/brainstorming
    ├── debugging          → ../skill-repos/superpowers/skills/debugging
    ├── tdd                → ../skill-repos/superpowers/skills/tdd
    ├── extract-notes      → ../skill-repos/dotfiles-skills/.claude/skills/extract-notes
    └── extract-notes-review → ../skill-repos/dotfiles-skills/.claude/skills/extract-notes-review
```

## Command Behavior

All commands operate on skillsets as the unit. `[[skill]]` is treated as backwards-compat syntax only.

### `install`

1. Read Skillfile, normalize `[[skill]]` entries to skillsets
2. One clone/fetch per skillset into `skill-repos/{skillset.name}/`
3. If `pin` exists, checkout that commit
4. For each skill in `skills[]`, create symlink: `skills/{skill}` → `skill-repos/{name}/{root_path}/{skill}`
5. Generate Skillfile.lock with flat `[[skill]]` entries + `skillset` field
6. Can also install from Skillfile.lock directly (groups by `skillset` field)

### `freeze`

1. Read Skillfile, resolve skillsets
2. One `rev-parse HEAD` per skillset (shared pin)
3. Write Skillfile.lock in flat format

### `update <skillset-name>`

1. Takes a skillset name
2. If given an individual skill name that belongs to a multi-skill skillset, error: `"brainstorming" is part of skillset "superpowers". Run: skill-set update superpowers`
3. Fetch and fast-forward merge the skillset's repo
4. Update pin in Skillfile.lock for all skills in that skillset

### `fork <skillset-name> <url>`

1. Takes a skillset name (same validation as `update`)
2. Change remote URL on the skillset's clone
3. Push to new remote
4. Update `origin` in Skillfile for that skillset

### `add` — dropped

Users edit Skillfile directly, then run `skill-set install`. Same as Bundler/Go modules workflow.

## Parsing & Normalization

### Skillfile → Skillset[]

- `[[skillset]]` entries map directly to `Skillset`
- `[[skill]]` entries normalize: `Skill { name, origin, path }` → `Skillset { name, origin, root_path: "", skills: [name] }` where `root_path` is derived such that `root_path + "/" + name == path` (or `root_path = ""` if `path == name`)

### Skillfile.lock → Skillset[]

- Read flat `[[skill]]` entries
- Group by `skillset` field to reconstruct `Skillset` objects
- Skills without `skillset` field → standalone single-skill skillset

## Migration

No migration. Old `Skillfile.lock` format (`[pins]` table) is broken by this change. Running `skill-set install` regenerates the lock file from the Skillfile in the new format. Users who had pinned commits in the old lock file will need to run `freeze` again after install.

`[[skill]]` entries in Skillfile continue to work unchanged.
